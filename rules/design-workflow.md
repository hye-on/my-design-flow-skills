# 디자인 워크플로우 규칙

> 디자인/UI 작업 시 자동 로드되는 규칙

## 토큰 사용 필수

- 새 UI를 생성할 때 반드시 `DESIGN.md`의 디자인 토큰을 사용한다
- 색상, 타이포그래피, 간격, 그림자 등 모든 시각적 속성은 토큰으로 표현한다
- 토큰이 없는 값이 필요하면 먼저 `DESIGN.md`에 토큰을 추가한 후 사용한다

## 색상 하드코딩 금지

- `#3B82F6`, `rgb(59, 130, 246)`, `hsl(217, 91%, 60%)` 등 하드코딩된 색상값 사용 금지
- 반드시 CSS 변수 (`hsl(var(--primary))`) 또는 Tailwind semantic class (`bg-primary`) 사용
- hook이 하드코딩된 색상을 감지하면 커밋이 차단됨

## 기존 컴포넌트 확인 우선

- 새 컴포넌트를 만들기 전에 반드시 기존 컴포넌트 확인:
  1. `components/ui/` -- shadcn/ui 기본 컴포넌트
  2. `components/` -- 프로젝트 커스텀 컴포넌트
  3. `knowledge-base/component-graph.md` -- 컴포넌트 그래프
- 이미 존재하는 컴포넌트와 유사한 것을 새로 만들면 hook이 경고함
- 기존 컴포넌트를 확장/수정하는 것이 새로 만드는 것보다 우선

## 4가지 상태 고려

모든 UI 컴포넌트/페이지는 다음 4가지 상태를 반드시 구현한다:

1. **로딩 (Loading)**: 스켈레톤 UI 또는 스피너
2. **에러 (Error)**: 에러 메시지 + 재시도 액션
3. **빈 상태 (Empty)**: 빈 상태 안내 + CTA
4. **성공 (Success)**: 데이터가 있는 정상 상태

## 다크모드 처음부터 고려

- 라이트모드만 먼저 만들고 나중에 다크모드를 추가하지 않는다
- 처음부터 semantic color token을 사용하면 다크모드는 자동으로 지원된다
- `bg-white`, `text-black` 대신 `bg-background`, `text-foreground` 사용
- 다크모드에서 이미지/아이콘이 보이는지 확인

## 한 플로우를 완전히 끝내고 다음으로

- 하나의 유저 플로우를 로딩 → 에러 → 빈 상태 → 성공까지 완전히 구현한 후 다음 플로우로 이동
- 반쪽짜리 UI 여러 개보다 완성된 UI 하나가 낫다
- 플로우 완성 기준:
  - 4가지 상태 모두 구현
  - 반응형 대응 완료 (Mobile, Tablet, Desktop)
  - 다크모드 대응 완료
  - 접근성 기본 요구사항 충족

## Knowledge Base 유지 규칙

- UI 파일 생성/수정 후 `knowledge-base/screen-inventory.md` 업데이트
- 컴포넌트 생성/수정 후 `knowledge-base/component-graph.md` 업데이트
- 디자인 토큰 변경 후 `knowledge-base/visual-language.md` 업데이트
- 변경 사항은 `knowledge-base/_sync-log.md`에 기록
- `SYNC_NEEDED` 플래그가 있는 파일은 다른 작업 전에 먼저 업데이트

### 선택적 도구 폴백 규칙
- 선택적 도구가 없으면 작업을 중단하지 말 것
- 작업 가능한 부분까지는 먼저 완료한다
- 도구가 필요한 시점에서 3가지 선택지(설치/직접/채팅)를 제시할 것
- 사용자가 선택하면 해당 경로로 진행
- 설치를 선택하면 설치 후 중단된 작업을 이어서 진행
- "없어서 못합니다"가 아니라 "없는데 어떻게 할까요?" 톤 유지

## 피드백 반영 시 토큰 레벨로 번역

피드백을 받으면 DESIGN.md 토큰 수준으로 번역하여 반영한다:

| 피드백 | 토큰 번역 |
|--------|----------|
| "이 색 좀 더 진하게" | `--primary` 값의 lightness 조정 |
| "버튼이 너무 작아" | 버튼 size variant의 height/padding 조정 |
| "글씨가 안 보여" | 해당 text 토큰의 font-size 또는 color contrast 조정 |
| "여백이 너무 좁아" | 해당 spacing 토큰 값 조정 |
| "전체적으로 둥글게" | `--radius` 토큰 값 조정 |

이렇게 하면:
- 피드백이 전체 시스템에 일관되게 반영됨
- 같은 피드백을 두 번 반영할 필요 없음
- 디자인 히스토리가 토큰 변경 로그로 추적 가능
